# 运营策略（电商运营诊断 · 第⑤段 · 收口）

> 链路终点。把上游三份诊断（详情页 / 人群 / 数据）综合成**一份按优先级排好、每条可执行、每条标来源**的运营动作清单。
> 铁律：**只用上游诊断里的事实，缺的别编；只给方向和动作，绝不预测业绩。**
> 口径源自工具代码 `ai/strategyPlanner.ts`。

---

## 触发

已经跑过上游某几段诊断，用户问"那接下来该打什么 / 怎么排优先级"。这是收口段——它**不产新诊断**，只把已有结论翻译成动作。

## 输入

- **产品情境**：品类 / 客单价 / 人群。
- **三份上游摘要**（有几份用几份）：
  - 【详情页诊断摘要】——五维哪些 red、最该补什么。
  - 【人群障碍摘要】——最致命障碍是哪条、卡住谁。
  - 【数据三联摘要】——黑洞渠道 / 死流量人群 / 王者 SKU。
- 缺哪份就标"未做（没有可用）"，**不基于它下结论**。

---

## 输出：按优先级排好的策略清单

```json
{
  "strategy": [
    {
      "dimension": "流量",
      "priority": "red",
      "action": "砍0转化的店播预算，转投出单渠道",
      "basis": "数据诊断：店播1000访客0转化=黑洞",
      "window": "30天"
    }
  ]
}
```

**铁律**：
1. **每条策略必须在 `basis` 写明它来自哪份诊断 / 哪条结论**（来源标注），不得凭空给建议。
2. **严禁预测业绩数字**（"GMV+X%""转化率提升X%"一律禁）——只给方向和动作。
3. **只用给到的三份摘要里的事实**，缺的维度别编；某份摘要为"未做"，就不要基于它下结论。
4. `dimension` 用简短词：流量 / 客群 / SKU / 详情页 / 内容 / 价格 / 复购。
5. `priority` ∈ red / yellow / green；`window` 写时间窗（如"30天"，没有就空串）。
6. 按优先级给 3-8 条，**红的排前面**，每条可执行。

---

## 输出收尾

- 策略清单（红 → 黄 → 绿）。
- 「上游缺口」：哪几份诊断没做 → 直链提示去补哪段（如"未做人群验证，建议先跑第③段再定客群动作"）。
- 「数据完备度 x/10」+「判断不了的」。

> 这一段是整链的出口：它的每一条 `basis` 都应能指回前面某段诊断的某条结论。指不回去的策略 = 凭空建议，删掉。
