# 详情页架构师

> 一个用方法论压住 AI 编造倾向的电商详情页工具。
>
> 核心约束：**没有数据不开工**。AI 不会假装替你做你应该做的市场调研。
> 它做的事情：把你给的真实数据，按方法论结构化输出成详情页方案。

---

## 🎯 这个 Skill 能做什么

| 你给我什么 | 我给你什么 |
|----------|----------|
| 5 项必填数据（见下方契约） | 完整 12-15 屏详情页方案（文案 + 配图要求 + 数据完备度评分） |
| 现成详情页的截屏 / 文案 | 逐项诊断报告 + 改进项优先级清单 |
| 现成文案 + 改写需求 | 按方法论标准重写的文案 |

---

## ❌ 这个 Skill 不会做什么（必须知道的边界）

- **不会自己抓详情页链接**：天猫/淘宝/京东全部有反爬+登录墙，Claude 抓不到内容，硬要"抓"只会基于 URL 编造产品信息。要诊断详情页 → 发截屏或文字。
- **不会预测转化率提升百分比**：LLM 没有数据支撑做不到这件事，凡是说能预测的都在骗你。
- **不会替你做市场调研**：四维分析需要的数据（搜索词、评价、竞品）必须你提供，AI 编造的数据没价值。

## 📋 启动前数据契约（强约束 · 不满足不开工）

> 这是 Skill 的核心硬约束。Claude 在执行**详情页生成任务**前，必须先完成数据契约校验。
> 校验不通过 → 不准开始生成，必须先要数据。

### 为什么要数据契约

详情页方法论的"四维分析法"依赖外部数据（竞品/搜索词/评价）。
没有这些数据，AI 只能凭通用品类常识编造，输出就是"看起来对但其实没数据支撑"的纸面分析。
强制契约的目的：让 AI 在**数据缺失时停下来**，不允许靠编填补。

### 🔴 必填项（缺一不开工）

| # | 项目 | 标准 | 不合格示例 | 合格示例 |
|---|------|------|-----------|---------|
| 1 | 产品品类 | 具体到子类，能识别使用场景 | "小家电" ❌ | "胶囊咖啡机（家用单杯/支持奶泡）" ✅ |
| 2 | 客单价 | 具体数字（元） | "便宜的" ❌ | "699 元" ✅ |
| 3 | 目标人群画像 | 年龄+城市+职业/身份+核心场景 | "上班族" ❌ | "25-35 岁一二线城市租房白领，工位上自冲咖啡省钱" ✅ |
| 4 | 核心参数 | ≥3 条具体规格/功能/材质 | "操作简单" ❌ | "19 Bar 高压萃取 / 30 秒出杯 / 0.6L 水箱可拆洗" ✅ |
| 5 | 竞品名录 | ≥3 个竞品名字 + 每个的主推卖点 | "市面上有很多" ❌ | "A 品牌主推咖啡馆品质；B 主推无线便携；C 主推性价比" ✅ |

### 🟡 强烈建议（缺了用 [推断] 填，报告会标"数据完备度"下降）

| # | 项目 | 标准 |
|---|------|------|
| 6 | 评价高频词 | 好评 + 差评各 3 个 |
| 7 | 用户搜索修饰词 | 主词后常加的 3-5 个修饰词 |

### 🟢 加分项（有则报告质量明显提升）

| # | 项目 |
|---|------|
| 8 | 生意参谋搜索词导出文本 |
| 9 | 评价区 30+ 条截图或文本 |
| 10 | 问大家高频问题 |

### 契约校验流程

```
用户说"帮我写一个 XX 详情页"
        ↓
Claude 出示「数据契约表」
        ↓
用户提供信息
        ↓
Claude 逐项校验必填 5 项
        ↓
   ┌────┴────┐
   ↓         ↓
有缺漏      全齐
   ↓         ↓
明确指出      校验强烈建议项
缺哪几项      ↓
不准开始    缺的标 [推断]
要数据      ↓
            开始生成 + 输出「数据完备度」分
```

### 校验失败时 Claude 的回复模板（强制使用）

```
我没法直接开始，缺这些必填数据：

❌ [列出缺失的必填项]
   缺失原因：[这一项的标准是什么]
   补法：[告诉用户怎么补]

数据齐全后我才会动笔。
方法论本身依赖数据驱动，缺数据硬填会出错。
```

### 数据质量门控（防止用户敷衍）

收到用户回复后，Claude 还要检查：

- **太宽泛**：人群=「上班族」→ 反问「具体什么年龄段、什么城市、什么使用场景？」
- **自相矛盾**：价格 = 99 元 + 目标人群 = 商务高端 → 反问「客单价和目标人群匹配性可疑」
- **数量不足**：竞品只列 1 个 → 「至少 3 个，否则无法做差异化分析」
- **没指标只情绪**：参数 = 「质量好」→ 「请列具体的功能/规格/材质参数」

只有**全部必填项 + 质量合格**才进入生成阶段。

### 数据完备度评分（每次报告必带）

```
─────────────────────────
📊 本次报告数据完备度：__/10
─────────────────────────
✅ 已用真实数据的章节：[列出]
⚠️ 用 [推断] 填充的章节：[列出]
💡 提升完备度的方法：[告诉用户下次该补什么]

如完备度 < 6：本报告偏向"通用品类指南"，
            决策前建议先把缺失数据补齐再重跑。
```

## 📷 素材获取规则（用于诊断现有详情页）

当用户要诊断**已有**详情页时，必须提供以下任一形式素材：

### A. 截屏（推荐）
从第 1 屏到最后一屏按顺序截图发我。我用视觉模型逐屏审查结构、卖点排序、文案因果公式、视觉规范。
- Windows：`Win + Shift + S`
- Mac：`Cmd + Shift + 4`

### B. 复制粘贴文案
适合纯文案诊断（看不到视觉部分）。

### 禁止行为（强约束）

- ❌ 试图用 WebFetch 抓天猫/淘宝/京东链接（拿到的都是登录页/反爬页）
- ❌ 基于 URL 路径猜产品内容
- ❌ 用通用品类知识硬填诊断
- ❌ 「无法访问，但根据你说的我猜...」这种半编造输出

拿不到素材就明确说**「无法诊断，需要截图或文案」**，等用户提供。

### 例外：小红书 / 微信公众号

用户装了对应 MCP（如 `xiaohongshu-mcp` / `weixin-reader`）时：
- 小红书链接 → 用 MCP 抓
- 微信公众号 → 用 MCP 抓

但仍走详情页诊断方法论。

## 🧠 核心方法论（详见 references/methodology.md）

### 1. 卖点挖掘四维法

每个维度必须标注数据来源（[数据] / [推断] / [用户原话]）：

- **维度一 对手对比**（必须基于用户提供的竞品名录）
- **维度二 搜索词**（用户提供 → [数据]，未提供 → [推断]）
- **维度三 评价痛点**（用户提供 → [数据]，未提供 → [推断]）
- **维度四 产品属性**（基于用户提供的核心参数）

### 2. 卖点排序优先级

```
差异卖点（人无我有 / 人有我优）  → 第 1-4 屏
   ↓
常见卖点（行业基本门槛）         → 第 5-8 屏
   ↓
购买保障（打消顾虑的承诺）       → 第 9-12 屏
```

### 3. 12-15 屏黄金结构

| 屏号 | 内容 |
|------|------|
| 第 1 屏（黄金屏） | 主标题（差异卖点 + 信任状）+ 配图 + 4 个证明点 + 促购引导 |
| 第 2-4 屏 | 围绕第一屏差异卖点展开（技术原理 / 使用场景 / 对比优势） |
| 第 5-8 屏 | 常见卖点与功能详解 |
| 第 9-12 屏 | 购买保障与用户见证 |
| 第 13-15 屏 | FAQ 与行动号召 |

### 4. 因果文案公式

**产品属性 + 解决方案**

❌ "面料柔软舒适" → 没有因果，是空洞描述
✅ "60 支长绒棉，像被云朵包裹" → 属性导出体验
✅ "19 Bar 高压萃取，30 秒出杯不等机器醒" → 属性导出场景

每屏标题 ≤ 8 字（超长换行最多 2 行），用具体数字而非模糊形容。

### 5. 视觉规范

- 背景色：单板块 ≤ 3 种，整页最多 2 种交替
- 避免红/灰/绿大面积背景（红色用于促销小标，灰色用于细节图）
- 字体：标题用粗体/艺术体（吸引）；正文用非衬线体（易读）
- 图片类型按品类选：服饰用模特图，功能性用场景图，标品用白底图

## ⚙️ 工作流程

### 流程一：从零生成详情页方案

```
1. 数据契约校验
   - 出示契约表
   - 收数据
   - 逐项校验 5 项必填
   - 缺漏则停下要数据（严禁靠常识编造）

2. 卖点挖掘（四维分析，每维标注数据来源）

3. 卖点分类与排序（差异 → 常见 → 保障）

4. 详情页结构规划（12-15 屏内）

5. 文案撰写（因果公式，每屏标题 ≤ 8 字）

6. 配图要求撰写（类型 + 画面 + 背景色 + 字体）

7. 自检清单逐项过

8. 输出数据完备度评分

9. 问用户：「这次有什么经验值得我记到经验库？」
   - 收到反馈用 Edit 工具追加到 data/经验库.md
```

### 流程二：诊断现有详情页

```
1. 收素材（截屏 / 粘贴文案，详见素材获取规则）

2. 按审查清单逐项检查
   - 内容结构（第一屏完整性 / 卖点排序）
   - 卖点有效性（两个问题检验）
   - 视觉呈现（背景色 / 字体 / 图片类型）
   - 文案因果公式（属性+解决方案）
   - 转化机制（限时限量 / 优惠券 / 问答）

3. 输出诊断报告
   - 综合评分 X/100
   - 优势项列表
   - 改进项按优先级（🔴严重 / 🟡中等 / 🟢轻微）
   - 改进项总数（不预测转化率提升幅度）
```

### 流程三：审查/重写现有文案

```
1. 收文案
2. 检查第一屏结构是否完整
3. 验证卖点排序逻辑
4. 检查每句是否符合因果公式
5. 给出问题清单 + 修改示例
```

## ✅ 自检机制

输出前必须勾完以下三张清单：

### 文案自检

- [ ] 每屏标题 ≤ 8 字（或最多 2 行）
- [ ] 使用「产品属性 + 解决方案」因果公式
- [ ] 差异卖点在前 3 屏聚焦展示
- [ ] 第一屏包含：主标题 + 信任状 + 4 证明 + 促购引导
- [ ] 总屏数控制在 12-15 屏内（低客单 6-8 屏例外）
- [ ] 数据具体（避免「很多」「非常」这种模糊表述）

### 视觉自检

- [ ] 每屏都有对应配图要求
- [ ] 图片类型符合品类规范
- [ ] 背景色符合规范（避免红/灰/绿大面积）
- [ ] 字体建议匹配产品调性

### 转化机制自检

- [ ] 限时限量提示
- [ ] 优惠券规则清晰（金额 + 期限 + 叠加）
- [ ] 问答列表减少咨询
- [ ] 购买保障充分（退换 / 质保）

发现问题立即修正，严重问题（如第一屏结构缺失）优先处理。

## 📚 经验库读写规则（强制执行）

> 本 Skill 的"跨会话记忆"通过一个**真实文件**实现，路径：
> `[本 Skill 目录]/data/经验库.md`
>
> 不是模型自动学习，是 Claude 用文件读写模拟出来的"记忆"。
> 只要遵守下面三条规则，第三次合作时确实会越来越懂用户的偏好。

### 规则 1：启动时必读

执行任何详情页任务前，第一步必须：

```
Read [本 Skill 目录]/data/经验库.md
```

读完后：
- 如该品类在 §2 有记录 → 优先用经验库里的卖点和偏好
- 如 §1 有用户风格偏好 → 按记录的风格生成
- 如 §3 有相关优秀案例 → 引用作为参考
- 如 §4 有相关失败教训 → 避免重蹈
- 全空 → 走通用方法论，但任务结束后必须写入

### 规则 2：完成时必写

任务结束后主动问用户：

> 「这次有哪些经验值得我记到经验库里？比如：你认可的卖点、不喜欢的表达、要避免的错误。」

收到反馈后，用 **Edit 工具**（不是 Write）追加到对应章节：

| 反馈类型 | 写入位置 |
|---------|---------|
| 用户偏好（风格/字数/严格度） | §1 用户风格偏好 |
| 新品类的卖点/痛点 | §2 品类卖点库 |
| 用户认可的优秀案例 | §3 优秀案例收藏 |
| Claude 这次做错的事 | §4 失败教训 |
| 待回填问题 | §6 待回填问题 |
| 上线后真实数据（转化/各屏停留/退款理由） | §7 投后效果回流（见 `数字分身/Skills/_电商投后回流口.md`） |

写入后告诉用户：「已记录到经验库 §X」

### 规则 3：引用历史时显式说明

应用历史经验时**明示来源**，例如：

> 「根据经验库 §1 你偏好直白风格，这次没用文艺修辞。」
>
> 「经验库 §4 记录过你不接受'极致'这种词，已替换成具体数字。」

引用历史的目的是让用户知道 Skill 真的在用经验库。

## 🎓 不同场景适配

### 低客单价产品（< 50 元）

- 简化结构 6-8 屏
- 重点突出性价比和基础功能
- 主图 + 标题即可促成转化

### 高客单价产品（> 200 元）

- 完整 12-15 屏
- 强化信任状（专利、检测报告、媒体背书）
- 增加用户见证和第三方背书
- 犹豫周期长，需充分打消顾虑

### 服饰类

- 必须模特图
- 强调面料、版型、穿搭场景
- 提供尺码建议和试穿报告

### 功能性产品（小家电/工具/护理）

- 强化功能参数标注（功率、容量、认证）
- 增加场景化视觉
- 优先级：安全性 > 功能性 > 颜值 > 易用性 > 性价比

### 美妆/个护

- 强调成分 + 使用感受 + 效果对比
- 用户实拍图重要性 > 模特图
- 需明确标注「个体差异」免责

## 🔧 增补层（本地·维特根斯坦补丁）

> 六刀审核结论：本 skill 已自带补丁①雏形（[数据]/[推断] 标注）、补丁⑤雏形（经验库）。
> 但三处需补强；刀③（文案打不打动人）不在此补——由"接力人群验证引擎"在系统层堵。
> 理论见 `数字分身/方法论/竞争策略框架的维特根斯坦边界.md`。

### 补丁③：卖点排序强制双解读（治刀⑥·把"看作"当"看见"）
"差异卖点排第 1-4 屏"是一种 *看作*。进前三屏的那个差异卖点，必须并列写反向读法 + 表态：
```
排第一的差异卖点：全尺码 XL-XXXL   [我赌用户最在乎这个]
⇄ 反读：也可能用户更在乎"价格"，全尺码只是加分项
赌的理由：竞品评论"大码买不到"高频（用户原话）
赌错代价：黄金第一屏喂错卖点，点击/停留全废
```
只对前三屏的差异卖点做，不每屏都加。

### 补丁①强化：综合评分挂置信、禁伪精确（治刀②·评级冒充测量）
诊断的 X/100 是判断不是测量。已在"诊断报告输出结构"焊入：分数后挂「置信·联动完备度」，完备度 < 6 声明仅供方向。永不把 78 分当成量出来的。

### 补丁⑤强化：经验库 ≠ 投后效果回流（治刀⑤残留·用过去假装未来）
现有经验库记的是**你的口味**（偏好/失败教训），不是**市场的反应**。
已加 §7 投后效果回流：上线后真实数据（转化/各屏停留/退款理由）回来，校验当初卖点排序判对没 → 倒改 §2 品类卖点库。
> 接 `数字分身/Skills/_电商投后回流口.md`——那才是"市场说话"的器官，本 skill 的卖点排序在那里被真实数据校验。
